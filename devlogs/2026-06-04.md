## Phase 1 — Core MVP

### Stack decisions
- **React Native + Expo SDK 56** over bare RN — EAS builds, managed config, no manual Xcode maintenance for most things.
- **Supabase over Firebase** — the whole reason is PostGIS. `ST_DWithin()` on a `GEOGRAPHY(POINT)` column gives native radius queries in metres. Firebase needs GeoHash workarounds and makes the credibility math painful to express in queries.
- **Expo Router v5** (file-based) over React Navigation — typed routes, no manual navigator setup, deep links handled automatically by the file structure.
- **NativeWind v4** for styling — Tailwind utility classes in RN, avoids a StyleSheet explosion across 10+ screens.
- **Zustand** for global map state (spaces cache, user location, filters). TanStack Query v5 for server state (profiles, reports, leaderboard).

### Auth
Google OAuth via Supabase Auth. Onboarding gate lives in `(tabs)/_layout.tsx` — reads `AsyncStorage` for an `onboarded` flag so new users always hit the intro screen before the map, regardless of deep link entry.

### Map and markers
`react-native-maps` with `PROVIDER_GOOGLE`. Custom map style strips POI labels and transit lines — cleaner backdrop for parking markers. Markers coloured by probability: green (≥65%), amber (35–65%), red (<35%). `ParkingMarker` is a custom callout view so we control shape and size exactly.

### Critical gotcha: Supabase Realtime + PostGIS WKB
**This cost ~2 hours.** Supabase Realtime ships `GEOGRAPHY` columns as WKB hex strings in the payload, not as GeoJSON. So `payload.new.location.coordinates` is always `undefined` — markers rendered at (0, 0).

Fix: added `STORED` generated columns `lat DOUBLE PRECISION GENERATED ALWAYS AS (ST_Y(location::geometry))` and same for `lng` to the `spaces` table (and `saved_locations`). These come through verbatim in both `select('*')` and Realtime payloads. Never try to parse `location` client-side.

### Report flow
- `submit_report` RPC does everything server-side: auth check, rate limit (5/hour, reset on hour boundary), GPS proximity (30m for new spaces, 100m for existing), 10-min duplicate guard per user per space.
- Pin dedup: before creating a new space, the RPC queries for an existing space within 25m. If found, the report is attached to it instead. Prevents map clutter from multiple pins on the same physical spot.
- `recompute_probability()` is called at the end of `submit_report` — time-decay weighted average (half-life ~1 hour) so probability is always fresh over Realtime without depending on a webhook.

### Credibility system
`profiles.credibility` 0–100, five tiers: 新手 (0–30, 0.5× weight) → 普通 (31–60, 1×) → 可靠 (61–80, 1.5×) → 達人 (81–99, 2×) → 守護者 (100, 3×). Weight capped by `report_weight_cap` which the `pattern-detection` Edge Function quietly reduces for persistent bad actors.

### Consensus windows
30-minute windows open when a report is submitted. `score-consensus` Edge Function scores them: if the next reported status matches the reported one, `confirmed = TRUE` and credibility goes up; if it contradicts, credibility goes down.

## Phase 2 — Intelligence

### Per-space pattern chart
`get_space_pattern(p_space_id)` RPC returns avg probability bucketed by hour-of-day (Taipei TZ, `AT TIME ZONE 'Asia/Taipei'`) for the last 30 days. Rendered as 24 bars in space detail. The current hour gets a border highlight. A "best time" insight is derived from whichever bar has the highest avg.

### Analytics tab
`get_area_pattern()` does the same aggregation across all spaces within 500m of the user. New 4th tab. Shows: current-moment card (what's the typical probability right now), hourly chart, best/worst 3 hours, and static tips.

### Heatmap toggle
`react-native-maps` `Heatmap` component only works on Android with Google Maps provider — not iOS. Used `Circle` overlays instead: each space gets a circle with `fillColor = PROBABILITY_COLORS(p) + '55'` (hex alpha). Works identically on both platforms. Markers remain tappable underneath by rendering them in a separate pass.

### Push infrastructure
`push_tokens` and `saved_locations` tables. `lib/notifications.ts` registers the Expo push token on sign-in and wires the notification tap handler to `router.push('/space/<id>')`. `notify-nearby` Edge Function queries `users_to_notify()` RPC — users with a saved location within the space's radius, notifications on, who aren't the one who just reported. **Requires a DB webhook wired in the Supabase Dashboard. Without it, nothing fires.**

## Phase 3 — Community

### Space notes
`notes TEXT` column on `spaces`. Passed as `p_note` to `submit_report` (optional param). Editable post-hoc via `update_space_note()` RPC — guards that `credibility >= 40` before writing.

### Verified lots
`verified BOOLEAN DEFAULT FALSE` on `spaces`. `submit_report` checks after every insert: if `COUNT(DISTINCT user_id) WHERE credibility_snap >= 70 >= 5` for this space, sets `verified = TRUE`. Marker gets a `check-decagram` overlay icon.

### Flag system
`space_flags` table, unique on `(space_id, user_id)`. `flag_space()` RPC upserts (so a user can change their reason). At 3 flags, the space's probability is reset to 0.5 to surface ambiguity.

### Streak
`last_report_date DATE` and `streak_days INT` on profiles. Inside `submit_report`: if `last_report_date = CURRENT_DATE - 1`, increment streak; otherwise reset to 1. Comparison uses Taipei TZ date.

## Phase 4 — Power

### Deep links
`scheme: "yo-parking"` was already in `app.json`. `Share.share({ url: 'yo-parking://space/<id>' })` on the detail screen. Expo Router's file-based routing handles `yo-parking://space/abc` → `app/space/[id].tsx` automatically.

### Directions
`Linking.canOpenURL('maps://app?daddr=lat,lng')` — opens Apple Maps if available, falls back to `https://maps.google.com/?daddr=lat,lng` otherwise.
