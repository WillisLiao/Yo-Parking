## Phase 5 — Discovery

### Nearby list view (`app/nearby-list.tsx`)
New screen accessible from a "列表" chip in the map toolbar. Shows all spaces within 1km sorted by distance (default) or probability. Three filter tabs: all / empty-only / bookmarks. Each row: probability circle, status pill, notes preview (1 line), time-ago, report count, distance. Tapping navigates to the existing space detail screen.

Kept this as a pushed screen rather than a tab — the map is still the primary surface, the list is an alternate view for users who prefer it. Adding it as a 5th tab would crowd the tab bar.

### Bookmarks
`bookmarked_spaces` table — simple `(user_id, space_id)` primary key. `toggle_bookmark()` RPC does the upsert/delete toggle and returns `{ bookmarked: boolean }`. Client invalidates `['bookmarked', id]` and `['bookmarks']` queries on success.

Bookmark icon lives in the top-right of space detail (above the probability circle). Also appears as a "收藏的車位" quick-access row on the profile screen, and as a filter tab in the list view.

`is_bookmarked()` RPC is a lightweight `EXISTS` check used to initialise the icon state when space detail mounts.

### Address search
Search icon added to the map top bar. Tapping it swaps the title/legend row for an inline `TextInput`. On submit (return key or search button), calls `Location.geocodeAsync(query)` and animates `MapView` to the first result.

**Typecheck gotcha:** `Location.geocodeAsync` in Expo SDK 56 only takes 1 argument. An earlier draft passed `{ useGoogleMaps: Platform.OS === 'ios' }` as a second arg — that's an older API. Caught by tsc, fixed before commit.

Platform geocoding quality: on iOS it uses CLGeocoder (Apple's geocoding), on Android it uses Google. For Taipei addresses both are accurate enough. Could upgrade to Google Places Autocomplete later for live suggestions, but that needs a separate API key and UI for the dropdown — deferred to future ideas.

## Phase 6 — Community+

### Space comments
`space_comments` table: `id`, `space_id`, `user_id`, `body TEXT CHECK (1–200 chars)`, `created_at`. No edit — delete your own comment and repost if needed.

Comments live as a collapsible section at the bottom of space detail. The query (`get_space_comments()`) is only triggered when the section is expanded — lazy load so the main space detail fetch stays fast.

`add_space_comment()` RPC: rate-limited to 5 per user per space per day (prevents spam without needing moderation). Returns `{ error: 'too_many_comments' }` if over limit — handled gracefully in the UI with an alert.

Each comment bubble shows: avatar initial, display name, badge colour, time-ago, body. Own comments have a trash icon. `delete_space_comment()` RPC guards `user_id = auth.uid()` server-side — client can't delete others' comments even with a crafted request.

`get_space_comments()` joins `profiles` so we get `display_name` and `badge` in one query without a separate profiles fetch.

## Phase 7 — Gamification

### Achievements
9 achievements across 4 categories:

| Category | Achievements |
|---|---|
| Report count | 初次出擊 (1), 回報達人 (10), 百戰老手 (100) |
| Correct reports | 社區支柱 (50 confirmed correct) |
| Streak | 週週到 (7 days), 月不間斷 (30 days) |
| Credibility | 可靠市民 (≥61), 停車達人 (≥81), 守護者 (100) |

`check_and_award_achievements(p_user_id)` is a `RETURNS SETOF TEXT` function — returns the IDs of newly earned achievements. It uses `INSERT ... ON CONFLICT DO NOTHING` for each check so it's idempotent and safe to call repeatedly. Returns only the IDs that were actually inserted this call, not all earned ones.

Called at the tail of `submit_report` v4: `SELECT ARRAY(SELECT check_and_award_achievements(v_user_id))`. The array is included in the JSON return value as `new_achievements`. The client checks this field on every report/confirm success and shows an inline alert if non-empty.

`get_user_achievements(p_user_id)` LEFT JOINs `achievements` with `user_achievements` — returns all 9 achievements with `earned_at NULL` for locked ones. The gallery screen (`app/achievements.tsx`) renders earned ones at full opacity with earned date, locked ones faded.

Profile screen shows a horizontal scroll of the first 6 achievements as pills, with a "查看全部 →" link.

### Daily missions
3 missions, defined in `daily_missions` table (static seed data):
- `report_empty` — report 1 empty space today
- `confirm_spaces` — confirm 2 existing spaces today  
- `explore_new` — create 1 new space today

Progress tracked in `user_daily_progress (user_id, mission_id, date, progress, completed)`. Date column uses `(NOW() AT TIME ZONE 'Asia/Taipei')::DATE` everywhere so the reset is at midnight Taipei time, not UTC.

Progress is updated inside `submit_report` v4 with three conditional `INSERT ... ON CONFLICT DO UPDATE` blocks — one for each mission, triggered by the relevant conditions (`p_status = 'empty'`, `NOT v_is_new`, `v_is_new`). This means mission progress updates atomically with the report, no separate API call needed from the client.

`get_daily_missions()` is the client-facing RPC — LEFT JOINs missions with today's progress for `auth.uid()`. Returns `progress`, `goal`, and `completed` for each mission. Profile screen queries this on mount and shows mission cards with progress bars.
